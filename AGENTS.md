# AGENTS.md — Guidance for AI Agents

This file describes the ApexCSS CLI codebase for AI agents. Read it before making any changes.

---

## Project Overview

**apexcss-cli** is a Node.js CLI tool (TypeScript, ESM) that builds, optimises, and manages the [ApexCSS](https://github.com/chris-briddock/apex) CSS framework. It compiles SCSS via Sass, tree-shakes unused CSS rules, caches builds, and provides a live-reload preview server.

- **Language:** TypeScript (strict mode, `noImplicitAny`, `noUnusedLocals`)
- **Module format:** ESM (`"type": "module"`)
- **Runtime:** Node.js ≥ 22
- **Linter/formatter:** Biome
- **Test runner:** Node.js built-in `node:test` via `tsx`
- **Package manager:** npm

## Key Commands

| Script | What it does |
|--------|-------------|
| `npm run build` | Compile TypeScript → `dist/` |
| `npm test` | Run all unit tests via `tsx tests/run-tests.ts` |
| `npm run test:coverage` | Tests + c8 coverage report |
| `npm run lint` | Biome lint on `cli/`, `bin/`, `tests/` |
| `npm run lint:fix` | Auto-fix lint issues |
| `npm run check` | Biome lint + format check |
| `npm run check:fix` | Auto-fix lint + format |

**Always run `npm run build && npm test` before finishing any change.** The build must be clean (zero TypeScript errors) and all tests must pass.

## Development Rules

### TypeScript
- Strict mode is on — no `any` without a `biome-ignore` comment explaining why.
- All imports use `.ts` extensions (NodeNext module resolution).
- No `declaration` or `declarationMap` — types live in `types/index.d.ts`.
- Avoid `noUnusedLocals` violations; remove unused imports before committing.

### Style (enforced by Biome)
- 2-space indentation, single quotes, no trailing commas, LF line endings.
- `const` over `let` wherever possible.
- Template literals over string concatenation.
- `for...of` over `.forEach`.
- Run `npm run check:fix` after any formatting change.

### Comments
- Write no comments unless the WHY is non-obvious (hidden constraint, subtle invariant, workaround).
- Never describe WHAT the code does — good names already do that.
- `biome-ignore` comments must include a colon and a reason, e.g.:
  `// biome-ignore lint/security/noGlobalEval: intentional — avoids compile-time type resolution for optional dep`

### File conventions
- One command per file in `cli/commands/`.
- Pure utilities (no side effects at import time) in `cli/utils/`.
- Each utility that is testable **must** have a corresponding test file in `tests/unit/utils/`.
- Tests use Node's built-in `node:test` + `node:assert` — no external test frameworks.

### Optional dependencies
- `lightningcss` is **optional** — do not add it to `dependencies` or `devDependencies`.
- Load it with `await import(specifier)` inside a try/catch; degrade gracefully if absent.
- To avoid TypeScript resolving optional module types at compile time, use a variable specifier with `eval()` or a type-only inline interface.

## Adding a New Command

1. Create `cli/commands/<name>.ts` with an exported `async function <name>Command(options)`.
2. Register it in `cli/index.ts` using `program.command(...)`.
3. Add tests in `tests/unit/commands/<name>.test.ts`.
4. Export any public types from `types/index.d.ts`.
5. Document new flags in `docs/LLMs.txt`.

---

## Adding a New Utility

1. Create `cli/utils/<name>.ts` — pure functions, no side effects on import.
2. Add tests in `tests/unit/utils/<name>.test.ts`.
3. Import with `.ts` extension: `import { foo } from '../utils/name.ts'`.

## Caching System

The build cache lives in `.apexcss-cache/index.json`. The cache key is SHA-256 of:
`configFileHash + allScssSourceHash + layers + minify + sourcemap`

On a cache hit, the build is skipped entirely. The cache is automatically invalidated when any SCSS source file changes. `apexcss build --no-cache` bypasses it.

Do not delete `.apexcss-cache/` manually — use `cacheClear()` from `cli/utils/cache.ts`.

## CSS Tree Shaking

`cli/utils/css-tree-shaker.ts` parses compiled CSS and removes rules whose selectors do not match any class found in the project's source files. It handles:
- Top-level rules
- `@media` blocks (responsive utilities)
- `@layer` blocks (CSS cascade layers)
- `@supports` blocks
- `@keyframes` / `@font-face` — always kept in full

The purge command integrates tree shaking via `--prune-css`. Always keep `shouldKeepSelector` conservative — false positives (keeping an unused rule) are safe; false negatives (dropping a used rule) break styles.

## CI / Release

- **CI** runs on every push/PR to `main`: lint, test (Node 22/23/24 matrix), build, smoke test.
- **Coverage** uploads to Codecov from the Node 22 job.
- **Publishing** runs only on `main` push, after the build job passes.
- **Releases** use `@changesets/cli`. Before merging a PR, run `npx changeset add` to document the change. The CI release workflow calls `changeset version` + `changeset publish`.

## What NOT to Do

- Do not commit `dist/` — it is gitignored and built by CI.
- Do not add `lightningcss`, `ws`, or other optional tools to hard `dependencies`.
- Do not use `process.exit()` inside utilities — only in command handlers.
- Do not skip `npm run build` before declaring a change done — TypeScript errors are invisible until compile time.
- Do not use `--no-verify` or `--no-gpg-sign` when committing.
- Do not use `console.log` directly — use `logger.*` from `cli/utils/logger.ts`.
- Do not hardcode paths — always use `resolve(cwd, ...)` relative to `process.cwd()`.
